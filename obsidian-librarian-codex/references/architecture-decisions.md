## ADR-001：方案 C（文件桥 + Agent 自动初始化）

**决策日期**：2026-06-26
**状态**：已定稿，v0.1 实施

### 背景

扩展（浏览器内）和 Hermes Agent（命令行）之间需要通信。三种方案被评估：

| 方案 | 通信机制 | 用户安装步骤 | 致命点 |
|------|---------|-------------|------|
| A | Python Bridge 常驻 HTTP 服务（127.0.0.1:8765） | 装扩展 + 双击 .pkg + macOS Gatekeeper 弹窗 + launchd 注册 | Gatekeeper 劝退 80% 普通用户；需要 $99/年苹果开发者证书签名；维护 3 套安装包 |
| B | Native Messaging（按需启动 Python） | 装扩展 + 双击安装包注册 native host manifest | 仍需 .pkg 签名；用户「跟 Agent 说装一下」破坏使用心流；用户得知道「能说什么」 |
| **C** | **本地文件桥 + Hermes cronjob 轮询** | **在 Hermes 里用一次（自动初始化）+ 拖扩展（一次性 30 秒）** | **隐式行为需要透明度补救；首次自检慢 5-10 秒** |

### 决策

**采纳方案 C**。

### 理由

1. **唯一符合「电灯哲学」的方案**——用户不「装」，是「发现它已经在了」
2. **不需要苹果/Windows 代码签名**（节省 $99 + $200/年）
3. **不需要打包 .pkg / .msi / .deb**（节省维护成本）
4. **不监听任何端口**，安全审计简单
5. **Agent 自检失败时可以友好提示**，比系统对话框可读性高
6. **首次用户体验**：用户用 Hermes 时顺手就把环境弄好了，没有「先安装再使用」的断点

### 通信契约

文件桥目录 `~/.obsidian-librarian/`：

| 子目录 | 写入方 | 读取方 | 用途 |
|--------|-------|-------|------|
| `inbox/{ts}-{nanoid}.json` | 扩展 | watcher | 入库任务 `{url, type, page_meta}` |
| `cookie/douyin.txt` | 扩展 | 抖音脚本 | cookie 字符串（权限 600） |
| `status/{task_id}.json` | watcher / Agent | 扩展 | 入库进度（下载中/分析中/写入中/完成/失败） |
| `handshake/extension.json` | 扩展 | Agent 自检 | `{version, last_seen}` 心跳（每 60 秒） |
| `extension/` | Agent 初始化 | 用户拖入浏览器 | 扩展源码包 |
| `archive/` | watcher | — | 已处理任务归档 |
| `failed/` | watcher | — | 失败 3 次后归档 |

### 用户物理动作总数

**全生命周期 = 2 个**：
1. 第一次在 Hermes 里随便用一次 obsidian-librarian（任何指令）
2. 拖一次扩展到 chrome://extensions/

之后零物理动作，永久无感。

### 透明度补救

方案 C 的弱点是「Agent 偷偷做了一堆事」。补救：

- 首次自检完成时，Agent **主动汇报**：做了什么、文件在哪、如何卸载
- 之后自检全通过时**静默**，不刷屏

### 未来路径

- v0.1：方案 C，依赖 Hermes
- v1.0（可选）：兜底加方案 A 的安装器，覆盖非 Hermes 用户。方案 C 始终保留作为 Hermes 用户的快捷通道

## ADR-002：用户唯一物理入口 = Chrome 扩展

**决策日期**：2026-06-26
**状态**：已定稿

### 禁止破坏的约束

不要让用户：
- 打开命令行 / 终端
- 编辑配置文件 / yaml / json
- 输入路径
- 跟 Agent 说「装一下 obsidian-librarian」（破坏心流）
- 看 README 学使用方法

### 用户路径

```
任何使用场景：
  发抖音链接 / 问知识库问题 / 要求维护

Agent 静默走自检：
  环境/目录/cronjob/vault/扩展包 全部就绪？
    是 → 直接干用户要的事
    否 → 自动修复能修的，提示拖扩展（如果缺）

用户拖扩展（仅一次）：
  chrome://extensions/ → 开发者模式 → 拖目录

之后永久无感。
```

## ADR-003：用户偏好硬约束（来自 USER.md）

**决策日期**：贯穿全项目

用户的产品哲学是硬性约束，AI 实施时**不允许妥协**：

1. **不要让用户接触命令行 / 配置文件 / 路径** — 所有交互通过 Chrome 扩展
2. **产品定位是「AI 时代基础设施」** — 像电灯，按开关就行
3. **项目结构服务于「AI 学的脚手架」** — references/ 记录设计决策给未来 AI 看
4. **零用户操作 = 完美** — 用户做的每一步都要可被砍掉，砍不掉的才保留

实施时反问自己：**这一步需不需要用户去碰终端？** 是 = 错的形态，重新设计。

## 评估新方案时的判定流程

未来如果有人提议「换一种通信方式」「加一个安装步骤」，按以下问题判定：

1. **用户物理动作数会从 2 增加吗？** 是 → 拒绝
2. **会让用户碰命令行/配置/路径吗？** 是 → 拒绝
3. **会破坏「在 Hermes 里用一次就自动配好」的心流吗？** 是 → 拒绝
4. **会引入苹果/Windows 代码签名成本吗？** 是 → 拒绝（除非项目盈利覆盖成本）
5. **会监听端口吗？** 是 → 慎重，需新的安全审计
6. **新方案的「魔法度」是否需要透明度补救？** 是 → 必须设计补救方案

任一拒绝条件触发 → 退回方案 C 原始设计。
