## 1. 通信方式：WebSocket

### 决策
使用 WebSocket（ws://127.0.0.1:8765）实现扩展与 Agent 实时通信。

### 原因
- 用户要求"立刻响应式回传"，文件桥有延迟
- 需要双向通信（扩展发请求，Agent 推状态）
- 比 HTTP 更实时，比 Native Messaging 更简单

### 替代方案
- **文件桥**: 已废弃，延迟高
- **HTTP POST**: 单向请求-响应，不能主动推送
- **Native Messaging**: Chrome 官方机制，但需注册 native host，复杂度高

### 后续风险
- Agent 需常驻运行（用户需保持启动）
- 断线重连逻辑需完善
- 防火墙可能拦截 WebSocket 端口

## 2. 扩展架构：控制塔

### 决策
扩展是「控制塔」：配置中心 + 状态看板 + Cookie 回传，不直接触发拆解。

### 原因
- 符合"电灯哲学"——用户按一下就行
- 拆解逻辑复杂（下载 + 分析 + 入库），放 Agent 端更灵活
- 扩展保持轻量，避免业务逻辑膨胀

### 替代方案
- **扩展直接触发**: 扩展调 API 拆解，但扩展能力受限（CORS、权限）
- **纯 Agent 触发**: 用户发链接给 Agent，但缺少可视化状态

### 后续风险
- 扩展和 Agent 需同时运行
- 状态同步有延迟（虽然 WebSocket 已最小化）

## 3. 视频拆解模型：doubao-seed-2-0-lite

### 决策
使用火山方舟 doubao-seed-2-0-lite-260428 作为主拆解模型。

### 原因
- 复刻信息量最强（88/100）
- 能听音频（抽帧 + 音频转写 + 时间轴对齐）
- 成本适中（91秒视频 ≈ 0.1275 元）

### 替代方案
- **Plus**: 最稳交付（90/100），但成本翻倍
- **Mini**: 低成本（82/100），信息量不足
- **Flash**: 时间轴压缩硬伤（72/100），不适合复刻

### 后续风险
- 模型可能更新/下线
- 成本可能变动
- 长视频（>10min）需警告

---

## 4. Cookie 管理：扩展自动抓取

### 决策
扩展自动抓取抖音 Cookie，不走命令行。

### 原因
- 用户不接触命令行
- 扩展有 chrome.cookies API 权限
- 实时回传，无需手动复制

### 替代方案
- **手动复制**: 用户从浏览器开发者工具复制，体验差
- **文件桥写入**: 已废弃

### 后续风险
- Cookie 可能过期，需定期重新抓取
- 抖音可能改 Cookie 结构
- 多账号切换需支持

---

## 5. 项目形态：Skill + 扩展

### 决策
纯 Skill + Python 脚本 + 模板，不走 MCP/npm/Node.js/HTTP 服务/Native Messaging（v0.1）。

### 原因
- 最小化依赖
- 自包含轻量方案
- 用户只需装扩展，其他全自动

### 替代方案
- **MCP**: 增加复杂度
- **Node.js 服务**: 增加运行时依赖
- **Native Messaging**: 增加系统级配置

### 后续风险
- v0.x 可能需要 HTTP API（快捷指令）
- 扩展能力受限（不能访问本地文件系统）

---

## 6. 目录结构：项目内嵌 vendor

### 决策
内嵌 Evil0ctal/Douyin_TikTok_Download_API 的 crawlers/douyin/web/，方式 B 整目录复制。

### 原因
- 避免外部依赖变动
- 内存 patch 注入 cookie（不写 vendor 文件）
- 自包含，不依赖网络

### 替代方案
- **pip 安装**: 外部依赖，可能不兼容
- **git submodule**: 增加复杂度

### 后续风险
- vendor 更新需手动同步
- 内存 patch 可能随版本失效
